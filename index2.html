4. 비가역 설계 원칙

Core 4개만 유지

Task Scheduler, Dispatcher M1, State Manager M2, Orchestrator M3만 Core다. Gate/OCR/LSTP/Reefer/BMS는 항상 Extension으로 분리한다.

Adapter는 무상태 변환기

연결, 재연결, raw payload 보존, Canonical Event 변환만 수행한다. 완료 판정, 취소 분기, 재배차 판단은 금지한다.

Kafka는 이력/전파 전용

장비 지시는 gRPC/SDK/Adapter를 통한 동기 경로로만 보낸다. Kafka는 outbox 기반 비즈니스 이벤트와 read model에만 사용한다.

완료는 이중 확인

한쪽 신호만으로 완료 금지. DS/LD/GI/GO/MI/MO/RH/AH 모두 완료 증거 집합이 충족되어야 CM-JR001을 보낸다.

구현 금지 목록
  • UI가 STS/ARMG/FMS 장비 API를 직접 호출하는 구조
  • Adapter 내부에서 C02 허가, OCR 판정, Replan 분기 수행
  • AT-MR002, S01-ARPT 고주기 이벤트를 동기 DB INSERT만으로 처리하는 구조
  • 장비 원문 필드명을 Core 내부 도메인 모델의 영구 이름으로 그대로 사용하는 구조

5. 시스템/배포 아키텍처

Layer 1. Connectivity DTQC OPC-UA, ARMG OPC-UA, FMS MQTT, TOS REST, Gate REST. 연결관리/보안/원문로그 담당.
Layer 2. Application Core 4개 + Extension Service. 모든 업무 판단은 여기서만 수행.
Layer 3. Event Backbone Kafka KRaft. equipment.telemetry, task.lifecycle, job.progress, alarm.events, replan.events, dashboard.readmodel, ecs.gate.transaction, ecs.ocr.result, ecs.audit.log.
Layer 4. Storage & Observability PostgreSQL 16, Redis 7.2, Prometheus, Grafana, ELK, OpenTelemetry.
Layer 5. Presentation 운영 UI는 Read Model과 승인된 Core API만 사용.

12. Dispatcher 구현

Dispatcher는 “장비 선택”만 담당한다
작업 흐름을 만들지 않는다. 어떤 QC/AGV/ARMG 조합이 가장 적절한지만 결정해 Orchestrator에 전달한다.

ETW / ETA 계산

ETW = now + (avg_cycle_time_sec * remain_ratio) + ETW_BUFFER_SEC
remain_ratio = 1 - (current_phase_progress / total_phases)

congestion_factor = zone_occupancy > 0.8 ? 1.4 : 1.2
ETA = now + (path_distance / agv_speed) * congestion_factor
if battery_pct < 30:
    ETA += 60 seconds

선정 규칙

  • DS/LD: DTQC + AGV + ARMG 조합
  • GI/GO: ARMG만 선정
  • AGV: at_on=Y, workable=Y, battery_status != EMERGENCY
  • DTQC: pointOfWork 일치, wkr_sts=Y, exe_mod=A, eqp_ctl_mod=A, hoist_sts=SH, eqp_sts not in TB/OF
  • ARMG: 담당 block 일치, wkr_sts=Y, exe_mod=A, eqp_sts not in TB/OF

판정 규칙

  • |ETW-ETA| ≤ 30초: 확정
  • ETW < ETA: 더 빠른 AGV 재탐색
  • ETW > ETA: AGV 선대기 허용, ETA drift 감시
  • 배터리 30% 미만: ETA +60초
  • Zone 점유율 0.8 초과: congestion factor 상향

14. Protocol Adapter 구현

DTQC / ARMG OPC-UA Adapter

  • ECS는 OPC-UA Client, 장비는 Server
  • Subscribe 주기 100ms
  • 재연결 3000ms 지수 백오프, max retry 10
  • NodeId: ns=1;s=DTQC.{EQP_NO}.{field}, ns=1;s=ARMG.{EQP_NO}.{field}
  • raw payload는 ecs.audit.log 발행

FMS MQTT Adapter

  • Broker: EMQX 5.x, TLS 필수
  • ecs/agv/{atId}/task/send QoS2
  • agv/{atId}/task/status QoS1
  • agv/{atId}/location QoS0
  • QoS1 duplicate는 idempotent merge 처리

Adapter 공통 체크리스트

  • raw request/response와 canonical event를 둘 다 저장
  • msgId/cmdId/correlation_id를 유지
  • invalid payload는 버리지 말고 audit + alarm 후 DLQ 또는 quarantine queue로 이동
  • Core에 전달하는 이벤트는 장비별 구조가 아닌 canonical 구조여야 한다

15. Extension Service 구현

Gate Service

OCR, ISPS, 예약 검증 후 /core/wi/process 호출. GI/GO 완료 후 ECS-GATE001 clearance를 송신한다.

OCR Service

GI 2L 시점에 OCR 검증. confidence 95% 이상이면 MATCH, 미달이면 2회 재촬영 후 수동확인.

LSTP / Reefer / BMS

GO 슬롯 예약/해제, RF plug-in, 배터리 상태 갱신을 맡는다. 모두 Core API를 통해서만 연계한다.

Quay 완료 증거: QC-JR005 + J04-ARPT 6C + LoadCell
7

8
Yard 완료 증거: YC-JR005 + AT-JR001 COMPLETE → CM-JR001 1회 송신

18. GI / GO / GC 구현 절차

GI

  • Gate가 OCR≥95% + ISPS + 예약 검증 완료 후 /core/wi/process 호출
  • Dispatcher는 ARMG만 배차
  • J07 재고조회 → XT 도착 대기(60분) → C02-AODR truckId 검증 → C02-TRPT → J06-TODR
  • 2L 시점 OCR 결과 대기
  • 완료: YC-JR005 + J06-ARPT 6C + OCR MATCH

GO / GC

  • GO는 appointment_time - 30분 시점에 LSTP reserve + J05-TODR preposition
  • 4M 상태에서 XT 대기 후 /core/xt/arrived 수신
  • ARMG C02-AODR.ITV_XT_NO == WI.truckId 일치 시만 C02-TRPT
  • 완료: YC-JR005 + J05-ARPT 6C, 이후 slot release
  • GC는 상태에 따라 즉시 취소 / C06-TODR / J01 복귀 분기
