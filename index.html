00 — Overview

ECS Core 개요

ECS(Equipment Control System) Core는 부산항만공사 진해신항 자동화 터미널의 심장부로, TOS의 작업 지시를 받아 DTQC·ARMG·AGV 등 현장 장비에 대한 실시간 배차·제어·모니터링을 담당하는 4개 핵심 컴포넌트의 집합입니다.

핵심 설계 원칙: ECS Core = 4개 컴포넌트만 (Scheduler·Dispatcher·State Manager·Orchestrator). Gate/OCR/LSTP/Reefer/BMS는 Extension Service로 분리. 장비 제어는 반드시 Protocol Adapter 경유. Kafka는 이력·감사·전파 전용 (장비 제어 아님). GI/GO는 DTQC·AGV 미사용.

01 — Architecture

ECS Core 컴포넌트 구조

4개 컴포넌트는 동기 gRPC로 연결되며, 비동기 이벤트는 Kafka로 발행합니다.

COMPONENT 01
Task Scheduler
ecs-scheduler / Module 0
TOS로부터 CM-JS001 WI를 수신하여 정합성을 검증하고 Internal Task를 생성합니다. 우선순위 큐(Redis Sorted Set)에 등록하고 Dispatcher에 배차를 요청합니다. GI/GO의 경우 Gate Service의 /core/wi/process 호출도 처리합니다.
WI 수신 검증 Task 생성 우선순위 큐
COMPONENT 02
Dispatcher
ecs-dispatcher / Module 1 (M1)
State Manager에서 장비 상태를 조회하여 ETW(QC 완료 예측)와 ETA(AGV 도착 예측)를 계산합니다. 두 값을 매칭하여 최적의 AGV와 ARMG를 선정하고 Orchestrator에 배차 결과를 전달합니다. GI/GO는 ARMG만 배차합니다.
ETW 산출 ETA 산출 최적 배차 AGV 선정
COMPONENT 03
State Manager
ecs-state-manager / Module 2 (M2) · SSOT
OPC-UA·MQTT Adapter에서 수신한 텔레메트리(100ms/200ms)를 정규화하여 Redis에 SSOT(Single Source of Truth)로 저장합니다. 장비 상태 이상을 감지하여 Orchestrator에 알림을 전송합니다. Kafka equipment.telemetry 발행도 담당합니다.
SSOT(Redis) 정규화 이상감지 텔레메트리
COMPONENT 04
Orchestrator
ecs-orchestrator / Module 3 (M3)
배차 결과를 받아 시나리오별 Job을 단계별로 분해합니다. Adapter를 통해 장비에 명령(J0x-TODR·AT-JS001)을 발행하고 ARPT/JR 보고를 수신합니다. Handshake를 처리하고 완료 조건을 이중 확인하며, 이상 발생 시 Replan Engine을 구동합니다.
Job 분해 Command 발행 Handshake Replan
컴포넌트 간 통신: Scheduler ↔ Dispatcher ↔ State Manager ↔ Orchestrator 는 모두 gRPC (동기)로 통신합니다. Kafka는 이벤트 발행(비동기) 전용이며 컴포넌트 간 제어 신호로 사용하지 않습니다. 장비 제어 명령은 반드시 Protocol Adapter를 경유합니다.

    ETW(Expected Time of Work-ready) 산출

    remainRatio = 1.0 - (currentPhaseProgress / totalPhases)
    remainTime = avgCycleTimeSec × remainRatio
    ETW = now() + remainTime + ETW_BUFFER_SEC // +30s
  • 2
    ETA(Estimated Time of Arrival) 산출 — AGV별
    각 AGV의 현재 위치(AT-MR002) · 현재 속도 · QCTP까지의 경로 거리 · Zone 혼잡도 보정계수를 적용
    pathDist = calcPathDistance(agv.pos, QCTP)
    travelSec = pathDist / agv.speed
    congestion = zoneOccupancyRate > 0.8 ? 1.4 : 1.2
    ETA = now() + travelSec × congestion
  • 3
    ETW/ETA 비교 → 3가지 경로 분기
    |ETW − ETA| 값에 따라 세 가지 처리 경로로 분기
    delta = abs(ETW - ETA)
    if (delta <= ETW_ETA_TOLERANCE_SEC) → 최적 매칭 → 배차 확정
    elif (ETW < ETA) → QC 먼저 완료 → 더 빠른 AGV 재탐색
    else (ETW > ETA) → AGV 먼저 도착 → POW 대기 + ETA Drift 감시
  • 4
    배차 확정 → gRPC DispatchJob 응답
    선정된 AGV_NO·ARMG_NO·DTQC_NO와 ETW·ETA를 Orchestrator에 gRPC 응답으로 전달. Task.status = SCHEDULED
    DispatchResult { dtqcNo, agvNo, armgNo, etw, eta }
    // GI/GO: agvNo = null (AGV 미사용)
GI/GO 배차 간소화: GI·GO 시나리오는 ETW/ETA 매칭 없이 ARMG만 배차합니다. 담당 Block(toSlot 기반) ARMG가 IDLE 상태이면 즉시 배차 확정. AGV 배차 로직 완전 생략.

// 완료 조건 상태 추적 (Redis per taskId)
class CompletionTracker {
  boolean condition1 = false  // YC-JR005 or QC-JR005
  boolean condition2 = false  // J0x-ARPT 6C or AT COMPLETE

  void onYcJr005(taskId) {
    condition1 = true
    checkBothConditions(taskId)
  }

  void onArpt6C(taskId) {
    condition2 = true
    checkBothConditions(taskId)
  }

  void checkBothConditions(taskId) {
    if (condition1 && condition2) {
      updateTaskStatus(taskId, COMPLETED)
      sendCmJr001ToTos(taskId)                    // TOS 완료 보고
      publishKafka("task.lifecycle", COMPLETED)   // Kafka 발행
      if (isGateScenario) sendEcsGate001()        // 출차 허가
    }
  }
}
