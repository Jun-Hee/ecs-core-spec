00 — System Concept

ECS란 무엇인가? — 자동화 터미널의 두뇌

ECS(Equipment Control System)를 처음 접하는 개발자를 위한 핵심 개념 설명입니다. 이 섹션을 먼저 이해하면 나머지 모든 내용이 명확해집니다.

자동화 터미널이란? 부산항만공사(BPA) 진해신항은 사람이 직접 크레인을 운전하지 않는 완전 자동화 컨테이너 터미널입니다. 컨테이너선이 접안하면 안벽크레인(STS)이 자동으로 컨테이너를 선박에서 꺼내고, 자율이송장비(AGV)가 야드까지 운반하며, 야드크레인(ARMG)이 지정된 위치에 쌓습니다. 이 모든 장비들을 실시간으로 조율·제어하는 시스템이 바로 ECS입니다.
📋
TOS는 "무엇을" 지시
TOS(Terminal Operating System, OPUS)는 컨테이너 계획 시스템입니다. "컨테이너 MSKU1234567을 선박 Bay 25에서 꺼내 야드 A01-03-B-2에 놓아라"와 같은 작업 계획을 수립합니다. TOS는 "어떤 장비로", "언제", "어떤 순서로"는 결정하지 않습니다.
⚙️
ECS는 "어떻게" 실행
ECS는 TOS의 작업 지시(WI)를 받아 실제 장비를 최적으로 제어합니다. 가장 빠른 AGV를 선택하고, STS가 컨테이너를 내릴 타이밍에 맞춰 AGV를 대기시키며, ARMG에게 정확한 위치에 적치하도록 명령합니다. ECS는 밀리초 단위의 실시간 장비 제어를 담당합니다.
🏭
장비들은 "명령대로" 실행
STS(안벽크레인), ARMG(야드크레인), AGV(자율이송장비)는 ECS의 명령을 받아 물리적 작업을 수행합니다. 각 단계 완료 시 ECS에 진행 상태를 보고하고, ECS는 이를 기반으로 다음 명령을 내립니다. 장비들은 자체 판단 없이 ECS의 시퀀스 제어를 따릅니다.
터미널 레이아웃 이해 — 컨테이너의 이동 경로
핵심 용어 사전 — 개발 시 반드시 알아야 할 용어
용어/약어전체명설명 (개발자 관점)
TOS / OPUSTerminal Operating System컨테이너 계획·재고 관리 시스템. ECS에 WI(Work Instruction)를 전송하고 완료 보고(CM-JR001)를 받는다. ECS 관점에서 클라이언트이자 상위 시스템.
STS / DTQC / AQCShip To Shore Crane안벽크레인. 선박↔플랫폼 컨테이너 이동. ECS가 OPC-UA로 J02/J04/J05/J07-TODR 명령 전송. 상태는 S01-ARPT, 진행은 J0x-ARPT로 보고.
ARMG / ATC / AYCAutomated Rail Mounted Gantry야드크레인. 야드 블록 내 컨테이너 적치·반출. ECS가 OPC-UA로 J01~J07-TODR 명령 전송. GI/GO 시나리오의 핵심 장비.
AGV / AT / FMSAutomated Guided Vehicle / Fleet Mgmt자율이송장비. STS 아래서 ARMG TP까지 수평 이송. ECS가 MQTT QoS2로 AT-JS001(배정)/AT-JS002(갱신)/AT-JS003(취소) 전송.
WIWork InstructionTOS→ECS 작업 지시(CM-JS001). 컨테이너 번호·작업유형(DS/LD/GI/GO 등)·출발/도착 위치 포함. ECS Core의 모든 처리는 WI 수신으로 시작.
WSTP / QCTPWater Side Transfer PlatformSTS 아래 AGV가 정차하여 컨테이너를 싣고 내리는 위치. DS 시 STS가 컨테이너를 WSTP AGV 위에 내려놓음.
LSTPLand Side Transfer Platform외부 트럭(XT)이 정차하여 ARMG와 컨테이너를 교환하는 위치. GI(반입) 시 XT가 LSTP에 정차하면 ARMG가 야드로 이동.
XT / OTExternal Truck / Over-the-road외부 도로 트럭. GI(Gate In 반입) · GO(Gate Out 반출) 시나리오에서 등장. truckId로 식별하며 truckId 검증이 안전의 핵심.
ETWExpected Time of Work-readySTS가 컨테이너를 플랫폼에 내려놓을 예측 시간. Dispatcher가 AGV 배차 시점 결정에 사용.
ETAEstimated Time of ArrivalAGV가 QCTP에 도착할 예측 시간. ETW와 ETA를 매칭하여 최적 AGV 선정.
C02-AODR/TRPTChassis Order/Transfer Report장비(STS/ARMG)가 AGV 또는 XT를 감지했을 때 ECS에 알리는 Handshake. ECS는 C02-TRPT로 정차 위치를 확인시켜줌.
Cmd_IdCommand IDECS가 장비에 작업 지시 시 생성하는 UUID. 장비는 모든 보고 메시지에 CMD_ID를 포함. Replan 시 새 CMD_ID 발급.
01 — Business Flow

시나리오별 터미널 운영 흐름

BPA_ECS_Biz_flow_v0.3 기반. 각 시나리오에서 어떤 장비가 어떤 순서로 작동하는지 전체 흐름을 이해합니다.

개발자 핵심 인식: ECS Core는 각 시나리오를 Phase(단계)로 분해하여 처리합니다. Phase 완료 조건이 충족되어야 다음 Phase가 시작됩니다. 완료 조건은 반드시 2가지 신호가 동시에 수신되어야 합니다 (이중 안전 확인). Orchestrator M3가 각 Phase의 상태를 추적하며 장비 명령을 순차 발행합니다.
DS (Discharge / 양하) — 선박에서 야드로
DS 개요: 선박의 컨테이너 → STS(선박→플랫폼) → AGV(플랫폼→야드 입구) → ARMG(야드 입구→적치 위치). 총 4단계 순차+병렬 처리.
Phase 1 STS Main Trolley — 선박→플랫폼 J02-TODR (EQP_TP=M)
① ECS → STS: J02-TODR 전송 (EQP_TP=M, 컨테이너 정보, Bay/Row/Tier)
② STS → ECS: QC-JR001 (Accept), 작업 수락 확인
③ STS 진행 보고: J02-ARPT/0M(이동시작) → 1M(픽업위치도착) → 2L(Twist Lock: 컨테이너 집음) → 3M(플랫폼방향이동) → 4M(플랫폼도착, 내려놓기 준비)
⚡ Phase 2(AGV배차)는 Phase 1과 동시에 병렬 시작. ETW/ETA 매칭으로 STS 완료 시점에 AGV가 도착하도록 조율.
Phase 2 AGV 배차 — QCTP 이동 (Phase 1과 병렬) AT-JS001 (taskType=QCDS)
① Dispatcher: ETW 산출 (STS 완료 예측) / ETA 산출 (AGV별 도착 시간 예측)
② ECS → FMS: AT-JS001 전송 (atId=AT301, taskType=QCDS, locTp=QCTP)
③ AGV → ECS: AT-JR001 APPROACH_QC (30m 전) → ARRIVE_LANE_ENTRY1 (QC 레인 도착)
⚡ ARRIVE_LANE_ENTRY1 수신 = Phase 3 트리거 조건
Phase 3 STS Portal Trolley — 플랫폼→AGV 상차 C02-AODR → J04-TODR (EQP_TP=S)
① STS → ECS: C02-AODR (ITV_NO=AGV번호, 차량 감지 신호)
② ECS → STS: C02-TRPT (AGV 정차 위치 확인, POS_CTNT=Buffer-01)
③ ECS → STS: J04-TODR (EQP_TP=S, Portal Trolley: 플랫폼→AGV 섀시 하차)
④ STS 진행: J04-ARPT/0M→1M→2L→3M→4M→5R→6C
⑤ AGV → ECS: AT-JR001 WORKING (LoadCell 값 포함: 컨테이너 실제 하중 확인)
🔴 Quay 완료 조건 (이중 확인): QC-JR005 (jobDone) AND J04-ARPT/6C AND LoadCell 확인. 셋 다 충족 시만 Phase 4로 진행.
Phase 4 ARMG — AGV에서 야드 적치 C02-AODR → J03-TODR (FM=WSTP, TO=야드슬롯)
① AGV → ECS: AT-JR001 APPROACH_YARD → ARRIVE_YARD_ENTRY1 (야드 입구 도착)
② ARMG → ECS: J07-AODR (야드 슬롯 재고 조회) → ECS → ARMG: J07-TRPT (재고 현황 응답)
③ ARMG → ECS: C02-AODR (ITV_XT_NO=AGV번호: ARMG-AGV Handshake, 경로B)
④ ECS → ARMG: C02-TRPT (WSTP 정차 위치 확인)
⑤ ECS → ARMG: J03-TODR (FM_PSN_CTNT=WSTP위치, TO_PSN_CTNT=야드슬롯A01-05-B-3)
⑥ ARMG 진행: J03-ARPT/0M→1M→2L→3M→6C
🔴 Yard 완료 조건 (이중 확인): YC-JR005 (ARMG jobDone) AND AT-JR001 COMPLETE (AGV 완료). 둘 다 수신 시 CM-JR001 → TOS 완료 보고.
LD (Load / 적하) — 야드에서 선박으로
LD 개요: DS의 역순. ARMG(야드→WSTP 상차) → AGV(WSTP→STS 플랫폼) → STS Portal(AGV→플랫폼) → STS Main(플랫폼→선박).

04 — How It Works

ECS Core 컴포넌트별 동작 원리 — 실제 터미널 운영 반영

각 컴포넌트가 실제 터미널 운영에서 어떤 판단을 하고 어떤 효과를 내는지를 개발자 관점에서 설명합니다.

Task Scheduler — ECS의 문지기 (입구 역할)

Task Scheduler는 TOS로부터 WI(Work Instruction)를 받는 ECS의 유일한 공식 입구입니다. "어떤 컨테이너를 어디서 어디로" 라는 TOS의 요청을 받아 ECS 내부 처리 단위인 Internal Task로 변환합니다.

실제 터미널에서의 역할: 하루에 수백~수천 개의 WI가 쏟아집니다. Task Scheduler는 HOT_JOB(긴급 작업)은 최우선으로, 나머지는 선박 출항 시간·우선순위 점수로 정렬하여 Redis Sorted Set에 관리합니다. Dispatcher는 이 큐에서 순서대로 꺼내 배차합니다.

Task 우선순위 계산 로직
// priority_score가 높을수록 먼저 처리
priority_score =
  (WI.priority × 1000)           // TOS 설정 우선순위 (1~999)
  + (선박출항잔여시간역수 × 100)  // 출항 임박할수록 높음

// HOT_JOB_FL=Y인 경우: 최전방 배치
if (WI.HOT_JOB_FL == "Y") {
  score = Long.MIN_VALUE + receivedAt
  // 사실상 큐 맨 앞으로
}

// Redis에 저장
redis.zadd("task_queue", score, taskId)
개발 주의사항: containerWiRef가 ECS 내 모든 Task 추적의 기준 키입니다. WI_UPDATE나 CANCEL 수신 시 이 값으로 기존 Task를 찾습니다. GI/GO WI는 Gate Service의 POST /core/wi/process로도 수신되므로 동일한 처리 로직을 공유해야 합니다.
Dispatcher M1 — 최적화 엔진 (ETW/ETA 매칭)

Dispatcher는 ECS의 배차 두뇌입니다. State Manager에서 전체 장비의 실시간 상태를 조회하여, STS가 컨테이너를 플랫폼에 내려놓는 시간(ETW)과 AGV가 도착하는 시간(ETA)을 예측하고 매칭합니다. 이 매칭이 정확할수록 STS와 AGV가 서로 기다리는 시간이 줄어 터미널 처리량(throughput)이 극대화됩니다.

State Manager M2 — 장비 상태의 단일 진실 소스 (SSOT)

State Manager는 터미널 내 모든 장비의 실시간 상태를 Redis에 유지합니다. STS는 100ms마다, AGV는 200ms마다 상태를 보내며, State Manager는 이를 정규화하여 표준 형식으로 Redis에 저장합니다. Dispatcher와 Orchestrator는 항상 State Manager를 통해서만 장비 상태를 조회합니다.

이상 감지의 핵심: 10초 이상 S01-ARPT가 수신되지 않으면 CRITICAL 알람. AGV의 workable=N이 되면 Task 상태에 따라 일반 Replan(T-02) 또는 Rescue Protocol(T-03)을 즉시 트리거합니다.

Orchestrator M3 — 시퀀스 실행자 & 완료 검증관

Orchestrator는 Dispatcher로부터 배차 결과를 받아 실제 장비 명령을 순서대로 발행합니다. 각 Phase의 완료 조건을 감시하고, 이중 확인(YC-JR005 AND J06-ARPT/6C 등)이 충족되면 다음 Phase로 진행합니다. 이상 발생 시 Replan Engine을 구동하여 대체 장비를 배차하거나 Rescue Protocol을 발동합니다.

Orchestrator가 호출하는 Extension Services:
• OCR Service: GI 시나리오에서 J06-ARPT/2L 수신 즉시 컨테이너 번호 검증 요청 (카메라가 ARMG Spreader에 설치)
• LSTP Service: GO 사전배치 시 LSTP 슬롯 예약, GO 완료 또는 GC 시 슬롯 해제
• Gate Service: GI/GO 완료 후 완료 통보 → Gate Service가 ECS-GATE001을 AutoGate에 전송하여 차단기 개방
• Reefer Service: GI 완료 + cntrTp=RF 시 냉동 플러그인 지시
C02 Handshake — 3가지 경로 상세 설명

C02-AODR는 장비(STS 또는 ARMG)가 차량을 감지했을 때 ECS에 알리는 메시지입니다. ECS는 발신 장비 유형과 ITV_XT_NO 필드 값으로 3가지 경로를 구분하여 처리합니다.

경로발신 장비ITV 필드시나리오ECS 처리후속 명령
경로 A STS (DTQC) ITV_NO = AGV번호 (ex: AT301) DS Phase3: STS Portal이 AGV를 감지 AGV 도착 확인 → C02-TRPT(Buffer 위치) → J04-TODR 발송 J04-TODR (Platform→AGV 하차)
경로 B ARMG ITV_XT_NO = AGV번호 (ex: AT301) DS Phase4: ARMG가 AGV를 WSTP에서 감지 AGV 도착 확인 → C02-TRPT(WSTP 위치) → J03-TODR 발송 J03-TODR (WSTP→야드슬롯 이적)
경로 C ARMG ITV_XT_NO = 트럭번호 (ex: TRUCK001) GI/GO: ARMG가 외부 트럭을 LSTP에서 감지 truckId 검증 필수! WI.truckId와 비교 → 불일치: CRITICAL 알람 + 작업중단 → 일치: C02-TRPT → J05/J06-TODR J06-TODR (GI) or J05-TODR (GO)
C02-AODR 경로 분기 판단 로직
fun handleC02Aodr(message: C02AodrMessage) {
  val eqpNo    = message.EQP_NO    // 발신 장비 번호
  val itvXtNo  = message.ITV_XT_NO  // AGV번호 또는 트럭번호

  when {
    isDtqcEquipment(eqpNo) → {
      // 경로 A: STS가 발신 → AGV Handshake (ITV_NO 필드 사용)
      handleAgvHandshake(message.ITV_NO, message)
    }
    isArmgEquipment(eqpNo) && isAgvId(itvXtNo) → {
      // 경로 B: ARMG가 AGV 감지 → ARMG-AGV Handshake
      handleArmgAgvHandshake(itvXtNo, message)
    }
    isArmgEquipment(eqpNo) && isTruckId(itvXtNo) → {
      // 경로 C: ARMG가 XT 트럭 감지 → truckId 검증 필수!
      if (itvXtNo != currentTask.truckId) {
        alarm.critical("TRUCK_ID_MISMATCH", itvXtNo, currentTask.truckId)
        send(C02Trpt(RSP_CD = "ERROR"))  // 거부!

    congestion = zoneOccupancyRate > 0.8 ? 1.4 : 1.2 pathDist = calcPathDistance(agv.position, QCTP) ETA = now() + (pathDist / agv.speed) × congestion if (agv.battery < BAT_LOW_THRESHOLD) ETA += BAT_LOW_ETA_PENALTY_SEC // BMS 연계
  • 3
    ETW/ETA 비교 → 3경로 분기
    delta = abs(ETW - ETA) if (delta <= ETW_ETA_TOLERANCE_SEC) → 최적 매칭 → 배차 확정 elif (ETW < ETA) → QC 먼저 완료 → 더 빠른 AGV 재탐색 else → AGV 먼저 도착 → POW 대기 + ETA Drift 감시
  • 4
    배차 확정 → gRPC DispatchJob(Orchestrator)
    orchestrator.dispatchJob(DispatchResult { dtqcNo, agvNo, armgNo, etw, eta // GI/GO: agvNo = null })

class CompletionTracker(taskId, jobTp) {
  boolean cond1 = false  // YC-JR005 또는 QC-JR005
  boolean cond2 = false  // J0x-ARPT 6C 또는 AT COMPLETE

  fun onYcJr005()    { cond1 = true;  check() }
  fun onArpt6C()     { cond2 = true;  check() }
  fun onAtComplete() { cond2 = true;  check() }  // DS Yard만

  fun check() {
    if (!cond1 || !cond2) return           // 둘 다 충족 시에만 진행
    updateTaskStatus(COMPLETED)
    tosAdapter.sendCmJr001(taskId)          // TOS 완료 보고
    kafka.publish("task.lifecycle", COMPLETED)
    if (jobTp in [GI, GO]) {
      gateService.notifyComplete(taskId)    // Gate Svc → AutoGate 출차허가
      if (jobTp == GO) lstpService.release(slotId)
    }
    if (jobTp == GI && cntrTp == RF) {
      reeferService.requestPlugin(cntrNo)   // 냉동 컨테이너 플러그인 지시
    }
  }
}

GC/CANCEL 상태별 처리 규칙
fun handleCancel(taskId: String, status: TaskStatus) {
  when (status) {
    PENDING, SCHEDULED →
      // A: 즉시 CANCELLED (장비 명령 없음)
      updateStatus(CANCELLED)

    AGV_ASSIGNED,
    QC_IN_PROGRESS,
    AGV_IN_TRANSIT →
      // B: 2L 이전 → C06-TODR 취소 경로
      if (currentPhase < PHASE_2L) {
        send(C06Todr(taskId))          // 크레인 취소
        send(AtJs003(agvNo))           // AGV 취소
        updateStatus(CANCELLED)
      } else {
        // C: 2L 이후 → J01-TODR 복귀 (컨테이너 제자리)
        send(J01Todr(fromSlot))        // 원위치 복귀
        lstpService.release(slotId)   // GO면 LSTP 해제
        updateStatus(CANCELLED)
      }

    COMPLETED →
      // D: 취소 불가 → TOS에 에러 응답
      throw BusinessException("ECS-CANCEL-001")
  }
}
