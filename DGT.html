4. DGT 고정 설계 원칙

Core 4개 유지

Task Scheduler, Dispatcher, State Manager, Orchestrator만 Core다. DGT 프로토콜 차이는 모두 Adapter에서 흡수한다.

외부 인터페이스는 DGT 실장비 기준

TOS JSON-RPC, STS Socket, ARMG REST를 원천 계약으로 삼고, 표준 문서는 canonical 참조로만 사용한다.

jobId는 ECS 생성

WI는 Plan 입력이며 실행키는 ECS가 만든다. 생성 즉시 TOS에 회신해 이후 모든 진행 인터페이스가 같은 키를 사용하게 한다.

Kafka는 내부 이벤트 버스

TOS와 장비를 Kafka에 직접 붙이지 않는다. Kafka는 DB outbox와 함께 Audit, UI, KPI, Replay 전용으로 한정한다.

DGT에서 반드시 제거해야 할 해석
표준 문서를 그대로 쓰면 STS/ARMG가 OPC-UA 장비처럼 보이지만, DGT 대상 개발에서는 이 해석이 틀린다. 실제 개발 계약은 표준 문서가 아니라 DGT 장비 인터페이스 정의서와 현장 메시지 방향이 우선이다.

5. DGT 목표 아키텍처

Layer 1. External Integration TOS JSON-RPC, SEOHO STS Socket, Rotem ARMG REST, FMS 인터페이스, Gate/OCR/BMS 확장연계.
Layer 2. Core Services Task Scheduler, Dispatcher, State Manager, Orchestrator. 내부 서비스 간 통신은 gRPC를 사용한다.
Layer 3. Internal Event Backbone Kafka는 task.lifecycle, job.progress, alarm.events, replan.events, dashboard.readmodel, ecs.audit.log 전용.
Layer 4. Storage PostgreSQL + Redis SSOT + Audit 저장. Work Instruction, internal_task, job_order, task_phase_history, event_outbox를 표준 유지하되 DGT sync 필드를 추가한다.
Layer 5. Presentation 운영 UI는 Read Model과 Core API만 호출한다. 장비와 TOS를 직접 치지 않는다.

DGT 프로토콜 맵

연계프로토콜설계 기준
TOS ↔ ECSJSON-RPCPlan 입력, Sync, 상태보고
ECS ↔ SEOHO STSSocketJ02/J04, QC-JR, C02, 상태 event
ECS ↔ Rotem ARMGREST APIJ03/J05/J06/J07, C02, 상태조회
Core ↔ CoregRPC배차, 상태질의, 실행요청
ECS 내부 비동기KafkaUI, 감사, KPI, 재생

왜 내부만 gRPC인가

  • Core 컴포넌트를 별도 컨테이너/Pod로 분리 배포할 수 있다.
  • RequestAssignment, DispatchJob, GetEquipmentState 같은 typed contract를 유지할 수 있다.
  • 외부 장비 프로토콜이 바뀌어도 Core는 흔들리지 않는다.
  • 따라서 gRPC는 장비 통신용이 아니라 마이크로서비스 내부 경계용이다.

7. TOS ↔ ECS JSON-RPC 설계

입력 원칙

  • WI는 실행 지시가 아니라 Plan 입력이다.
  • 한 번에 LIST 형태로 복수 작업을 전송할 수 있다.
  • 필수 정보는 선박/VVD, Bay/Row/Tier, cntrNo, jobTp, from/to slot, 컨테이너 속성이다.
  • 이 단계에는 jobId가 없다.

출력 원칙

  • ECS는 internal task 생성 시점에 task_id와 job_id를 만든다.
  • TOS에는 같은 상관키로 WI Sync를 되돌린다.
  • 중간 상태를 포함해 최소 5회 동기화한다.
  • TOS는 Kafka 대신 JSON-RPC 응답/통지로만 ECS 상태를 본다.

DGT JSON-RPC 기본 구조

{
  "jsonrpc": "2.0",
  "method": "setWorkInstruction",
  "id": "msg-uuid",
  "params": {
    "eventType": "WI_NEW | WI_UPDATE | CANCEL",
    "workInstructions": [
      {
        "containerWiRef": "WI-2026-0001",
        "jobTp": "DS",
        "cntrNo": "ABCD1234567",
        "fromBay": "25",
        "fromRow": "12",
        "fromTier": "06",
        "pointOfWork": "QC101"
      }
    ]
  }
}

8. ECS ↔ SEOHO STS Socket 설계

명령

J02-TODR, J04-TODR를 Socket frame으로 전송한다. 표준의 OPC-UA method call 개념은 Socket send/ack로 치환한다.

진행

J02-ARPT, J04-ARPT, QC-JR001~5를 모두 수신해 phase와 장비 응답을 이중으로 본다.

정위치

C02-AODR / C02-TRPT는 SEOHO 장비가 보내는 정위치 요청과 ECS 허가 응답으로 해석한다.

Phase 1에서 5R, 6C가 빠져 보이는 문제
표준 Biz Flow 일부 그림에서는 J02 단계가 0M~4M까지만 강조되지만, DGT 실제 운전에서는 포털/후속 단계와 결합되어 5R, 6C 의미가 사라지는 것이 아니다. 따라서 DGT 설계서에서는 J02, J04 각각의 phase map을 분리해 0M~6C를 명시하고, QC-JR 이벤트와 함께 저장해야 한다.

SEOHO Socket Frame 예시

{
  "msgNo": "J04-TODR",
  "msgId": "uuid",
  "cmdId": "uuid",
  "eqpNo": "QC101",
  "eqpTp": "S",
  "cntrNo": "ABCD1234567",
  "agvNo": "AT301",
  "laneNo": "2",
  "phase": "DISPATCH"
}

9. ECS ↔ Rotem ARMG REST 설계

핵심 원칙

  • J03/J05/J06/J07은 REST endpoint 호출로 정의한다.
  • C02 handshake는 REST request + callback 또는 polling response로 구현한다.
  • 야드 위치, TPW/TPL, 블록/베이/로우는 response body에 canonical location으로 다시 적재한다.

권장 Endpoint

  • POST /api/v1/armg/jobs/j03
  • POST /api/v1/armg/jobs/j05
  • POST /api/v1/armg/jobs/j06
  • POST /api/v1/armg/inventory/j07
  • POST /api/v1/armg/handshake/c02
  • GET /api/v1/armg/jobs/{cmdId}

11. jobId / taskId 정책

DGT 기준 확정
WI는 Plan 데이터이며 jobId가 없다. 따라서 ECS가 jobId를 생성하고, 이후 TOS·STS·ARMG·FMS에 동일 상관키 체계를 유지해야 한다.
1
WI 수신
containerWiRef를 멱등키로 사용한다. 아직 jobId는 없다.
2
Task 생성
Task Scheduler가 task_id와 scenario root job_id를 생성한다.
3
TOS Sync
CM-JR001 상당의 JSON-RPC sync로 ECS 생성 jobId를 TOS에 회신한다.
4
장비 명령 파생
STS, FMS, ARMG에는 장비별 cmd_id를 파생하되 root job_id와 연결한다.

12. Task Scheduler 재설계

DGT용 입력 처리 규칙

  • LIST형 WI를 batch ingest 한다.
  • 검증은 선박/Bay/Row/Tier, cntrNo, jobTp, pointOfWork, slot 형식, 컨테이너 속성까지 포함한다.
  • 같은 containerWiRef에 대해 WI_UPDATE / CANCEL을 상태별로 병합한다.
  • 검증 통과 즉시 task_id, job_id를 생성하고 TOS sync 대기열에 넣는다.
  • priority score는 표준 공식을 유지하되 선박 출항시간, Bay 집중도, 같은 QC work queue 밀집도를 반영한다.

13. State Manager 재설계

상태 수집원

  • SEOHO Socket status event
  • Rotem REST status callback/polling
  • FMS task/status/location
  • TOS sync ack

핵심 추가 항목

  • socket connected / disconnected 상태를 장비 health로 반영
  • REST timeout과 HTTP 5xx를 anomaly로 승격
  • TOS sync 실패를 business anomaly로 기록
  • AGV lane 할당 결과를 SSOT에 저장

14. Dispatcher 재설계

배정 시점

WI 수신 직후가 아니라 검증 완료 후. 초기 WI에는 agvNo, armgNo가 없다는 것을 전제로 한다.

DS 배정

QC는 pointOfWork 기준, AGV는 ETA/배터리/혼잡도 기준, ARMG는 Yard block 기준으로 선정한다.

GI/GO

현재 GOS 설계 부재이므로 DGT 1차 범위에서는 DS 파생 또는 ARMG 중심 단순 모델로 운영한다.

ETW / ETA 해석 주의

WI 단계에서 ETW/ETA는 아직 확정값이 아니다. DGT 설계에서는 장비 확정 이후 계산된 예측값으로 정의한다. 즉 ETW는 QC/ARMG의 예상 작업 가능 시점, ETA는 AGV가 해당 handshake 위치에 도착할 예상 시점이다.

16. DS 상세 시나리오

P0
Plan Intake
TOS가 WI LIST를 보낸다. 선박, 컨테이너, Vessel Bay/Row/Tier, 위치 정보가 포함된다. jobId는 없다.
P1
ECS Task 생성
Scheduler가 우선순위를 정하고 task_id/job_id를 만든 뒤 TOS에 CREATED sync를 보낸다.
P2
QC Main 작업
SEOHO STS로 J02-TODR를 보내고 QC-JR001, J02-ARPT 0M~4M을 추적한다.
P3
AGV Dispatch 병렬
Dispatcher가 AGV를 선정한 뒤 FMS task를 생성한다. Lane 정보는 AGV/FMS 응답으로 보정한다.
P4
QC Portal Handshake
J04-TODR 이후 SEOHO가 정위치 요청을 보내면 C02 승인 후 5L/6C까지 진행한다. 이때 QUAY_DONE sync를 보낸다.
P5
Yard Delivery
새로운 AGV 이동 또는 목적지 갱신 후 Rotem REST J03-TODR를 발행한다. APPROACH_YARD, ARRIVE, WORKING, COMPLETE 상태를 추적한다.
P6
종결
YC/J03 완료와 AGV COMPLETE가 함께 충족되면 YARD_DONE 및 COMPLETED sync를 순차적으로 보낸다.
DGT DS 최종 해석
STS에는 사실상 두 번의 의미 있는 작업지시(J02, J04)가 가고, FMS는 quay 접근/yard 이동을 포함한 AGV task를 관리하며, yard 측은 Rotem J03가 최종 적치 실행점이 된다. 따라서 표준 Biz Flow를 DGT에 맞추면 Quay 완료와 Yard 완료를 분리해 관리하는 것이 가장 안정적이다.

17. GI / GO / GC 적용 원칙

GI

현재 GOS 연동 설계가 부재하므로 1차 적용에서는 DS 파생으로 해석한다. 즉, 외부 차량 흐름을 제외한 core sequencing, slot 검증, ARMG 하차 논리를 DS-yard 변형으로 우선 구현한다.

GO / GC

표준 문서 구조는 유지하되 DGT 실제 장비 계약이 확보되기 전까지는 설계 flag로 격리한다. 운영 투입 시에는 Gate/외부차량/GOS 연동 상세 계약이 추가되어야 한다.

18. DB / Kafka / Audit 정책

DB

표준 DDL 골격은 유지한다. 단 DGT sync용 필드와 protocol_type, vendor_ack_code, tos_sync_seq를 추가한다.

Kafka

task.lifecycle, job.progress, alarm.events, replan.events, dashboard.readmodel, ecs.audit.log는 유지하되 TOS 직접 연동은 제거한다.

Audit

JSON-RPC request/response, Socket frame, REST request/response를 모두 원문으로 저장한다. 추후 현장 이슈는 이 로그가 기준이다.

DGT DB 확장 권장

ALTER TABLE internal_task
  ADD COLUMN IF NOT EXISTS root_job_id uuid,
  ADD COLUMN IF NOT EXISTS tos_sync_status varchar(30),
  ADD COLUMN IF NOT EXISTS quay_completed_at timestamptz,
  ADD COLUMN IF NOT EXISTS yard_completed_at timestamptz,
  ADD COLUMN IF NOT EXISTS dgt_phase varchar(50);

ALTER TABLE job_order
  ADD COLUMN IF NOT EXISTS protocol_type varchar(20),
  ADD COLUMN IF NOT EXISTS vendor_msg_no varchar(30),
  ADD COLUMN IF NOT EXISTS vendor_ack_code varchar(20);

19. 내부 gRPC / Proto 적용

DGT 외부 연동이 JSON-RPC, Socket, REST로 바뀌더라도 내부 proto와 gRPC 서비스는 유지한다. 이는 표준 bpa_ecs_core_v1.proto를 버리는 것이 아니라, 외부 Adapter를 갈아끼워도 Core 계약은 고정한다는 뜻이다.

Task Scheduler -> Dispatcher : RequestAssignment
Dispatcher -> State Manager : GetEquipmentState / GetAvailableEquipment
Dispatcher -> Orchestrator : DispatchJob
State Manager -> Orchestrator : NotifyAnomaly
